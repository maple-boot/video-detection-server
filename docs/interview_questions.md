## 题目一：从图片检测到视频流检测的跨越（考察经验迁移能力）

**题目：**

你的简历显示主要做的是**图片目标检测**。我们现在需要做的是**实时视频流检测**——接收 RTMP 直播流，逐帧检测后推流输出。

请说说你认为**图片检测**和**实时视频流检测**最大的区别是什么？从图片检测迁移到视频流检测，你会遇到哪些新的挑战？

**追问：**

1. 图片检测时你可能处理一张图等几秒没关系，但视频流是 25fps（每帧 40ms），推理如果要 80ms 你怎么处理？
2. 视频流中同一个目标会持续出现在多帧里，你怎么利用这个时间维度的信息？和单张图片检测的处理方式有什么不同？
3. 视频流可能因为网络问题中断，你的系统应该怎么应对？

**参考答案要点：**

| 差异维度 | 图片检测 | 视频流检测 |
|----------|----------|------------|
| 时效性 | 可以等几秒 | 必须实时（<40ms/帧） |
| 目标连续性 | 每张图独立 | 同一目标跨帧持续存在 → 需要跟踪 |
| 资源管理 | 处理完释放 | 长时间运行 → 内存泄漏、GPU 显存管理 |
| 容错 | 失败重试即可 | 流中断需要重连、看门狗、资源清理 |
| 输出 | 返回结果即可 | 需要实时推流 + 异步上报 |

**关键考察点：**
- 是否意识到推理速度是核心瓶颈 → 跳帧检测、TensorRT 加速
- 是否想到利用时间维度 → 目标跟踪（DeepSort/ByteTrack）
- 是否有工程化思维 → 容错、资源管理、长时间运行的稳定性

**评分标准：**

| 等级 | 表现 |
|------|------|
| 优秀 | 能系统对比两者差异，提到推理速度、跟踪、容错、资源管理等多个维度 |
| 良好 | 能提到推理速度和跟踪，但对容错和资源管理考虑不足 |
| 及格 | 只能想到"推理要快"这一个点 |
| 不及格 | 认为"视频就是多张图片"，没有意识到本质差异 |

## 题目二：DeepSort vs ByteTrack（考察目标跟踪的理解深度）

**题目：**

你的简历显示你用过 DeepSort 做目标跟踪。我们项目中用的是 ByteTrack。

请说说你对 DeepSort 的理解：它的核心流程是什么？用了哪些信息来做目标匹配？

然后请思考：如果要你把 DeepSort 换成 ByteTrack，你觉得两者的核心区别是什么？各自有什么优缺点？

**追问：**

1. DeepSort 中的"Deep"指的是什么？ReID 特征在什么情况下会失效？
2. ByteTrack 的核心思想是"不丢弃低置信度检测框"，为什么要这样做？这对跟踪效果有什么影响？
3. 在无人机航拍场景中（目标小、运动快、摄像头也在动），你觉得 DeepSort 和 ByteTrack 哪个更合适？为什么？

**参考答案要点：**

**DeepSort 核心流程：**
- 检测框 → 卡尔曼滤波预测位置 + reID网络特征库 → 匈牙利算法匹配 -> 更新特征库数据
- 匹配策略：级联匹配（总代价最小匹配）
- 匹配依据：马氏距离（运动信息）+ ReID 特征余弦距离（外观信息）

**DeepSort vs ByteTrack 核心区别：**

| 维度 | DeepSort | ByteTrack |
|------|----------|-----------|
| 外观特征 | 依赖 ReID 网络提取特征 | 不用外观特征，纯运动匹配 |
| 低置信度框 | 直接丢弃 | 保留并参与二次匹配（核心创新） |
| 计算开销 | 需要额外的 ReID 前向推理 | 无额外网络，更轻量 |
| 遮挡处理 | 靠 ReID 重新识别 | 靠低置信度框维持轨迹 |
| ID 切换 | ReID 特征相似时容易误匹配 | 运动模型一致时更稳定 |

**无人机航拍场景分析：**
- 目标小 → ReID 特征质量差（像素不够提取有效外观特征）
- 摄像头运动 → 运动模型需要更鲁棒
- ByteTrack 更合适：不依赖外观特征，利用低置信度框维持轨迹

**评分标准：**

| 等级 | 表现 |
|------|------|
| 优秀 | 能清晰描述 DeepSort 流程，准确对比两者差异，能结合场景分析选型 |
| 良好 | 知道 DeepSort 用了 ReID + 卡尔曼滤波，能说出 ByteTrack 不用外观特征 |
| 及格 | 知道 DeepSort 的基本流程，但对 ByteTrack 不了解 |
| 不及格 | 只能说出"DeepSort 就是检测+跟踪"，无法展开技术细节 |

## 题目三：YOLO + SAHI 切片检测与参数调优 ⭐（核心考察题）

**题目：**

我们的场景是**无人机航拍视频**，画面分辨率 1920x1080，目标（车辆、人员）在画面中占比很小。

直接用 YOLO 检测效果不好，因为模型输入是 640×640，原图缩放后小目标就只剩几个像素了。所以我们用了 **SAHI（切片检测）**：把大图切成多个 640×640 的小块，每块独立检测，最后合并结果。

**请回答：**

1. 你之前做图片检测时有没有遇到过小目标检测的问题？你是怎么处理的？
2. 切片检测有哪些潜在问题？怎么解决？

**然后看我们的配置：**

```yaml
model:
  default_conf: 0.75      # 推理置信度阈值
  report_conf: 0.8         # 上报置信度阈值
  max_box_ratio: 0.4       # 框占画面最大比例（过滤过大的误检框）
  imgsz: 640               # 模型输入分辨率

sahi:
  enabled: true
  slice_size: 640
  overlap_ratio: 0.1       # 切片重叠 10%
  iou_threshold: 0.5       # NMS 合并阈值

inference:
  batch_size: 8            # 批量推理大小
```

**三个线上问题，请逐一分析你会怎么调：**

1. **远处小目标漏检严重**（车辆、人员经常检测不到）
2. **误检率高**（地面阴影、水面反光被误识别）
3. **推理耗时过长**（SAHI 切片后延迟高，画面卡顿）

**追问：**

1. `default_conf` 和 `report_conf` 为什么要设两个不同的值？如果只用一个会怎样？
2. `overlap_ratio` 从 0.1 调到 0.3 会怎样？有什么代价？
3. `imgsz` 从 640 调到 1280，对小目标检测和推理速度的影响分别是什么？
4. 你之前用 TensorRT 加速推理，能说说 TensorRT 的工作原理吗？为什么比 PyTorch 快？
5. 如果 TensorRT engine 的 batch_size 是 4，但配置里设的是 8，你觉得会发生什么？

**参考答案要点：**

**小目标漏检：**
- 降低 `default_conf`（如 0.6），让更多低置信度目标被检出
- 增大 `imgsz`（如 640→1280），小目标像素更多更容易被检测
- 增大 `overlap_ratio`（如 0.1→0.3），减少切片边界截断目标
- 减小 `slice_size`（如 640→480），让小目标在切片中占比更大

**误检率高：**
- 提高 `default_conf`（如 0.75→0.85）
- 提高 `report_conf`（如 0.8→0.9），上报时更严格
- `max_box_ratio` 过滤：过大的框（整片阴影）应被过滤
- 训练数据增加负样本

**推理慢：**
- 增大 `batch_size`（充分利用 GPU 并行）
- 跳帧检测（不是每帧都推理）
- TensorRT 加速（候选人应该熟悉）
- 多 GPU 并行推理

**两个置信度阈值的原因：**
- `default_conf`(0.75)：推理时保留稍多的框，维持跟踪连续性（DeepSort/ByteTrack 需要低置信度框）
- `report_conf`(0.8)：上报时只要高置信度结果，保证业务方收到的结果质量

**TensorRT 原理考察：**
- 层融合（Layer Fusion）：合并相邻算子减少 kernel launch
- 精度校准：FP32→FP16/INT8 量化
- Kernel Auto-Tuning：针对目标 GPU 选择最优 kernel
- 动态 Batch：engine profile 支持 min/max batch 范围

**TensorRT batch_size 不一致：**
- engine 编译时指定了固定 batch 范围（如 min=1, max=4）
- 配置设 8 但 engine 只支持 max=4 → 需要分两次推理（代码中已有分 chunk 逻辑）
- 或者在导出 engine 时用更大的 batch 重新编译

**评分标准：**

| 等级 | 表现 |
|------|------|
| 优秀 | 能给出具体参数调整方案 + 理解参数间权衡 + 能深入解释 TensorRT 原理 |
| 良好 | 能回答大部分参数调整，对 TensorRT 了解但不够深入 |
| 及格 | 知道调 conf 和 imgsz，但对 SAHI 和 TensorRT 原理不清楚 |
| 不及格 | 只能给出"调低阈值"或"换大模型" |

## 题目四：多 GPU 推理与模型资源管理（考察工程化能力）

**题目：**

我们的服务器有多张 GPU。我们把 GPU 分成两组：
- **编解码 GPU**：专门用于 FFmpeg 视频解码/编码（NVDEC/NVENC）
- **推理 GPU**：专门用于 YOLO 模型推理（CUDA）

**Q1：** 为什么要这样隔离？编解码和推理放在同一张 GPU 上会有什么问题？

**Q2：** 如果推理 GPU 有 3 张，一个模型加载到所有 3 张卡上做推理，切片数量不同时（比如 2 个切片 vs 20 个切片），你怎么决定用几张卡？

**追问：**

1. 模型加载到 GPU 上后，显存被占满了，此时要加载另一个新模型怎么办？
2. 多个任务共用同一个模型，最后一个任务结束时能直接卸载模型吗？如果有其他任务还在用呢？
3. 你之前做 TensorRT 部署时，遇到过显存不足的情况吗？你是怎么处理的？

**参考答案要点：**

**GPU 隔离原因：**
- NVDEC/NVENC 和 CUDA 推理争抢同一张卡的显存和计算资源
- 视频编解码有固定的延迟要求，被推理抢占会导致解码卡顿
- 隔离后互不干扰，各自稳定运行

**动态 GPU 选择：**
- 切片少（<batch_size）→ 单卡就够，多卡反而有通信开销
- 切片多（>batch_size × GPU数）→ 分发到多卡并行，每卡处理一部分
- 关键原则：每张卡至少承担 batch_size 个切片，否则不值得启动额外 GPU

**模型资源管理：**
- **显存不足时**：驱逐最早加载的模型池（LRU 策略），释放显存后加载新模型
- **多任务共用**：引用计数（refcount），任务开始 +1，结束 -1，归零时才卸载
- **竞态条件**：加载/卸载需要加锁，避免并发操作导致模型被意外卸载

**评分标准：**

| 等级 | 表现 |
|------|------|
| 优秀 | 理解 GPU 隔离原因，能设计引用计数 + LRU 驱逐的资源管理方案 |
| 良好 | 理解 GPU 隔离，知道引用计数，但对动态调度和驱逐策略不清楚 |
| 及格 | 知道要分开用 GPU，但对资源管理缺乏设计思路 |
| 不及格 | 不理解为什么要隔离，或认为"多卡就是速度快" |

## 题目五：问题排查 — 线上告警分析（考察实际问题解决能力）

**题目：**

线上环境出现以下告警日志：

```
源流疑似中断 | 无帧间隔=3.5s | 阈值=2.0s | frame_count=12345
推理耗时过高 | gpu=2 | algorithm_id=alg_001 | batch_n=16 | time=350ms
```

用户反馈：推流画面出现**绿色花屏区域**，检测框不再更新。

请分析可能的原因，给出你的排查思路。

**追问：**

1. "源流疑似中断"说明了什么？为什么是"疑似"而不是"确认断开"？
2. 推理耗时突然从 50ms 飙到 350ms，可能的原因有哪些？
3. 花屏可能出在哪个环节？（解码？推理？渲染？推流？）你怎么定位？
4. 如果你能加一个调试功能帮助排查，你会加什么？

**参考答案要点：**

- **排查顺序**：源流 → 解码输出 → 推理 → 推流，逐层定位
- **"疑似中断"**：无帧间隔超过告警阈值(2s)但未超过强制断开阈值(30s)，中间状态
- **推理耗时升高**：GPU 显存不足 / GPU 降频 / 其他任务抢占 / batch 翻倍触发多次推理
- **花屏定位**：保存解码原始帧截图，对比解码输出和最终推流画面，确认哪个环节引入花屏
- **花屏原因**：FFmpeg 解码器在流中断恢复后 P/B 帧依赖断裂，输出损坏帧

**评分标准：**

| 等级 | 表现 |
|------|------|
| 优秀 | 能系统性分层排查，提到解码器 warmup、GPU 显存管理、中间帧调试 |
| 良好 | 能分析主要问题方向，但排查思路不够系统 |
| 及格 | 能猜到是流或 GPU 的问题，但无法展开 |
| 不及格 | 只说"重启服务"或"检查网络" |
